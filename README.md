<h1 align="center">📚 Conceitos Fundamentais de QA</h1>

<h2 align="center">Diferença entre Erro, Falha e Defeito</h2>

Se você trabalha com desenvolvimento de software ou engenharia de qualidade, provavelmente já ouviu falar nesses termos. Embora sejam usados de forma intercambiável, é importante compreender suas diferenças.

`ERRO` Uma ação humana que produz um resultado incorreto ou inesperado. Uma desenvolvedora pode cometer um erro ao escrever um código incorreto, uma testadora pode errar ao não cobrir determinada funcionalidade, e uma usuária pode inserir dados incorretos em um formulário.

`FALHA` A manifestação externa de um erro — quando o produto não executa conforme o esperado. Por exemplo, uma aplicação de pagamento que apresenta erro no processamento. As falhas podem ser detectadas pelas usuárias ou por testadoras durante o processo de testes.

`DEFEITO` A causa raiz da falha — o problema subjacente que resulta no comportamento incorreto. Se um código incorreto gera um erro de processamento de pagamento, esse código é o defeito. Defeitos podem ser introduzidos em qualquer fase: codificação, teste ou design.

> Compreender esses conceitos é fundamental para desenvolver e manter produtos de alta qualidade. Identificando e corrigindo defeitos, prevenimos falhas e garantimos que o produto funcione corretamente.

[Saiba mais sobre o tema: page 16](https://bstqb.online/files/syllabus_ctfl_4.0br.pdf)

<h2 align="center">Safety vs Security</h2>

A diferença entre `Safety` e `Security` é fundamental em engenharia de software e gestão de riscos.

**`Safety (Segurança Funcional)`**

`Definição:` Proteção contra falhas ou acidentes que podem causar danos às pessoas, ao meio ambiente ou ao sistema. Está relacionado à `confiabilidade` e à `integridade funcional`.

`Foco:` Prevenir riscos de erros **não intencionais** — falhas de hardware, bugs de software ou condições operacionais inadequadas.

Exemplos:
- Sistema de freios de um carro que deve evitar acidentes
- Software médico que controla doses e evita overdoses
- Controle de temperatura em usina nuclear

---

**`Security (Segurança da Informação)`**

`Definição:` Proteção contra `ataques intencionais` ou acessos não autorizados que comprometam confidencialidade, integridade ou disponibilidade de dados e sistemas.

`Foco:` Prevenir riscos de `ações maliciosas` — hackers, vírus, roubo de dados, espionagem.

Exemplos:
- Criptografia de dados sensíveis
- Autenticação de usuárias para evitar acessos indevidos
- Firewalls e sistemas de detecção de intrusões

---

**`Diferença Principal`**

| Aspecto | Safety | Security |
|----------|----------|----------|
| Foco | Prevenir danos por falhas não intencionais | Prevenir danos por ataques intencionais |
| Origem do Risco | Erros, falhas de hardware/software | Ataques cibernéticos, acessos não autorizados |
| Exemplo Prático | Avião não deve cair por falha no sistema | Sistema bancário não deve ser invadido |
| Objetivo | Sistema seguro e confiável em operação | Dados e sistemas protegidos contra ameaças |

> Em sistemas complexos como carros autônomos, `Safety` e `Security` estão interligados: um ataque de Security pode causar uma falha de Safety.

**Resumindo:**
- `Safety` = Proteção contra falhas **não intencionais**
- `Security` = Proteção contra ameaças **intencionais**

## Postman + Newman — Testes de API na Prática

O **Postman** é a ferramenta central da minha rotina de testes de API. Utilizo-o para validar endpoints REST — status codes, bodies de resposta, autenticação Bearer Token, variáveis de ambiente e scripts de teste automatizados.

O **Newman** complementa o fluxo rodando as collections diretamente via terminal e gerando relatórios em HTML.

**Instalação do Newman:**
```bash
npm i -g newman
```

**Rodando uma collection:**
```bash
newman run nome_do_arquivo.json
```

**Gerando relatório HTML:**
```bash
npm i -g newman-reporter-htmlextra
newman run nome_do_arquivo.json -r htmlextra
```

**Exemplos de validações que costumo fazer:**
- Criação de usuário com e-mail dinâmico via script
- Autenticação e geração de Bearer Token
- Registro, listagem e exclusão de recursos com e sem autenticação
- Validação de respostas com senha inválida (cenários negativos)

## 🚀 Testes de Automação com Cypress

O **Cypress** é minha principal ferramenta de automação. Trabalho com ele utilizando o padrão **Page Objects** e **BDD com Cucumber/Gherkin**, cobrindo cenários de regressão, testes funcionais e de API.

| Vantagem | Descrição |
|----------|----------|
| ✅ Execução rápida | Testes confiáveis e estáveis |
| ✅ JavaScript nativo | Linguagem que domino no dia a dia |
| ✅ BDD com Gherkin | Cenários legíveis para todo o time |
| ✅ CI/CD | Integrado ao pipeline de entrega contínua |
| ✅ Page Objects | Código organizado e reutilizável |

### Estrutura que utilizo nos projetos:

```
📁 cypress/
 ├── 📂 e2e/            # Cenários BDD em Gherkin (.feature)
 ├── 📂 fixtures/       # Massa de dados para os testes
 ├── 📂 support/
 │    ├── 📂 pages/     # Page Objects
 │    ├── 📂 steps/     # Step definitions (Cucumber)
 │    └── commands.js   # Comandos customizados
 ├── 📂 screenshots/
 ├── 📂 videos/
📄 cypress.config.js
```

### Exemplo de cenário BDD com Gherkin:

```gherkin
Feature: Login no sistema financeiro

  Scenario: Login com credenciais válidas
    Given que estou na página de login
    When preencho o e-mail "nathalia@empresa.com"
    And preencho a senha correta
    Then devo ser redirecionada ao dashboard
```

### Exemplo de Page Object:

```javascript
class LoginPage {
  visit() {
    cy.visit('/login')
  }
  fillEmail(email) {
    cy.get('#email').type(email)
  }
  fillPassword(password) {
    cy.get('#password').type(password)
  }
  submit() {
    cy.get('[data-cy=btn-login]').click()
  }
}
export default new LoginPage()
```

### Exemplo de Step Definition:

```javascript
import LoginPage from '../pages/LoginPage'

Given('que estou na página de login', () => {
  LoginPage.visit()
})
When('preencho o e-mail {string}', (email) => {
  LoginPage.fillEmail(email)
})
```

### Instalação rápida:

```bash
mkdir meu-projeto-cypress
cd meu-projeto-cypress
npm init -y
npm install cypress --save-dev
npx cypress open
```

<summary>⚙️ CI/CD — Integração e Entrega Contínuas</summary>

<br>

Tenho experiência na configuração de **pipelines CI/CD** integrados à suíte de automação com Cypress, reduzindo o tempo de validação por sprint e aumentando a confiança nos deploys.

### Exemplo de pipeline com GitHub Actions:

```yaml
name: QA Automation - Cypress

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  cypress-run:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout do repositório
        uses: actions/checkout@v3

      - name: Instalar dependências
        run: npm install

      - name: Executar testes Cypress
        run: npx cypress run

      - name: Upload de artefatos (screenshots/videos)
        uses: actions/upload-artifact@v3
        if: failure()
        with:
          name: cypress-artifacts
          path: |
            cypress/screenshots
            cypress/videos
```

SELECT * FROM transacoes

AND status = 'PROCESSADO'
ORDER BY data_criacao DESC;

-- Verificar integridade de integração com ERP
SELECT a.id, a.valor, b.status_erp
FROM lancamentos a
LEFT JOIN integracao_erp b ON a.id = b.lancamento_id
WHERE b.status_erp IS NULL;
